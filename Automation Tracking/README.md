# Automation Tracking (NinjaOne)

These scripts sync NinjaOne automation/scheduled-task activities into a local SQLite database, push per-device script status into a NinjaOne custom field, generate time-windowed HTML reports from that database, and generate monthly HTML dashboards published to the NinjaOne Knowledge Base.

## Scripts overview

| Script | Purpose |
|--------|---------|
| **Get-AutomationActivities.ps1** | Pulls NinjaOne activities (DEVICE/ACTION) via the API, enriches with device/org/location, and writes to a SQLite database. First run seeds the most recent 1,000 activities; subsequent runs are incremental using `newerThan`. Uses the **NinjaOneDocs** module for auth and API calls. Requires PowerShell 7. Uses **sqlite3.exe**: place it in the script directory, in PATH, or pass `-SqliteExePath`. Download from https://www.sqlite.org/download.html (Precompiled Binaries for Windows). |
| **Get-AutomationActivitiesByDays.ps1** | Retrieves NinjaOne activities (DEVICE/ACTION) for a configurable number of days (1–90) from the API and inserts them into the same SQLite database. Uses `after`/`before` epoch parameters and `olderThan` pagination. No duplicate entries: uses the same table and `INSERT OR IGNORE` by activity `id`. Same credentials, **sqlite3.exe**, and DB path as Get-AutomationActivities. Use for backfilling a specific day range (e.g. `-Days 45`). |
| **Invoke-ScriptStatusSync.ps1** | Reads from the same SQLite DB (latest COMPLETED run per device + script), maps result to SUCCESS/FAILURE, and writes a per-device map to a NinjaOne device custom field (e.g. `scriptStatus`). Requires PowerShell 7 and the NinjaOneDocs module. Uses **sqlite3.exe** (same as Get-AutomationActivities): place it in the script directory, in PATH, or pass `-SqliteExePath`. Download from https://www.sqlite.org/download.html. |
| **Invoke-ScriptTracker.ps1** | Reads from the same SQLite DB (Activities.db), filters by Past 10 Days using epoch-based date ranges, and generates HTML reports: AutomationHeadsUp (global success/failure summary at top, then one card per script showing most recent execution **per device**), per-automation detail pages for all automations (one row **per device** per script—most recent execution per device), and per-device detail pages for all devices (full history). When a KB article ID map (`.kb-article-ids.json`) is present, links to other report articles use NinjaOne deep links; run **Update-ScriptTrackerKbArticleMap.ps1** after Publish-ScriptTrackerToKnowledgeBase so the map is current. Uses the same credential sources (NinjaOneDocs, Ninja-Property-Get / env / params) and **sqlite3.exe**. Output uses inline styles only for NinjaOne KB. |
| **Update-ScriptTrackerKbArticleMap.ps1** | Builds or refreshes the `.kb-article-ids.json` mapping file (article name → id, parentFolderId) by enumerating global KB articles under the Script Tracking folder. Run after Publish-ScriptTrackerToKnowledgeBase.ps1 or on a schedule so Invoke-ScriptTracker.ps1 can emit correct deep links. Standalone; same credential resolution as other scripts. |
| **Publish-ScriptTrackerToKnowledgeBase.ps1** | Takes the HTML output from Invoke-ScriptTracker.ps1 and uploads it to NinjaOne as global Knowledge Base articles. Creates new articles or updates existing ones by name; skips updates when content is unchanged (hash-based, using a sidecar `.kb-article-hashes.json` file). Run after Invoke-ScriptTracker. Uses NinjaOneDocs and the same credential resolution. |
| **Invoke-AutomationDashboard.ps1** | Reads activities for a calendar month from the same SQLite DB, analyzes success/failure, and generates HTML dashboards per organization and one global dashboard, then publishes them as Knowledge Base articles under "Monthly Reports". Optional PowerShell 7 upgrade. |
| **Invoke-ActivitiesDbMaintenance.ps1** | Deletes rows from the Activities table older than a configurable retention (default 90 days). Use `-Vacuum` to reclaim disk space after deletion. Run on a schedule (e.g. weekly) to keep the database size under control. Uses **sqlite3.exe** (same as other scripts). |

## Recommended run order

1. **Get-AutomationActivities.ps1** — Run on a schedule (e.g. daily or hourly) so the SQLite database stays updated with recent activities. Optionally use **Get-AutomationActivitiesByDays.ps1** with `-Days` (1–90) to backfill a specific number of days from the API; it uses the same DB and does not create duplicate rows.
2. **Invoke-ActivitiesDbMaintenance.ps1** — Run on a schedule (e.g. weekly) to delete Activities rows older than the retention window (default 90 days). Use `-Vacuum` to reclaim disk space. Optional but recommended to prevent unbounded DB growth.
3. **Invoke-ScriptStatusSync.ps1** — Run after Get-AutomationActivities (or on the same schedule) to push the latest script status from the DB to NinjaOne device custom fields.
4. **Invoke-ScriptTracker.ps1** — Run after Get-AutomationActivities (e.g. daily or on demand) to generate time-windowed HTML reports from the DB; output is KB-friendly for use in NinjaOne articles. If `.kb-article-ids.json` exists (see Update-ScriptTrackerKbArticleMap.ps1), report-to-report links use NinjaOne deep links.
5. **Publish-ScriptTrackerToKnowledgeBase.ps1** — Run after Invoke-ScriptTracker to upload those HTML reports to NinjaOne as global KB articles (create or update by name; skips unchanged content by default).
6. **Update-ScriptTrackerKbArticleMap.ps1** — Run after Publish (or on a schedule) to refresh the article ID map so the next Tracker run can emit correct deep links. Optional but recommended when using KB linking.
7. **Invoke-AutomationDashboard.ps1** — Run monthly (e.g. first day of the month) to generate and publish the previous month’s automation dashboards to the NinjaOne KB.

## Prerequisites

- **NinjaOne API:** Client ID and Client Secret with appropriate scopes. Credentials are read via `Ninja-Property-Get` (ninjaoneInstance, ninjaoneClientId, ninjaoneClientSecret) when scripts run in a NinjaOne context, or must be set where the script runs.
- **NinjaOneDocs:** Required for Get-AutomationActivities, Get-AutomationActivitiesByDays, Invoke-ScriptStatusSync, Invoke-ScriptTracker, Update-ScriptTrackerKbArticleMap, Publish-ScriptTrackerToKnowledgeBase, and Invoke-AutomationDashboard. See https://github.com/lwhitelock/NinjaOneDocs.
- **SQLite:** Get-AutomationActivities, Get-AutomationActivitiesByDays, Invoke-ScriptStatusSync, Invoke-ScriptTracker, and Invoke-ActivitiesDbMaintenance use the **sqlite3.exe** command-line tool. Place sqlite3.exe in the script directory, add it to PATH, set `-SqliteExePath`, or install to `C:\RMM\sqlite3.exe`. Download from https://www.sqlite.org/download.html (Precompiled Binaries for Windows). The database file (default `C:\RMM\Activities.db`) is created by Get-AutomationActivities if it does not exist.
- **PowerShell 7:** Required for Get-AutomationActivities and Invoke-ScriptStatusSync; Invoke-AutomationDashboard can run on Windows PowerShell 5.1.

## Where to change paths and options

- **Get-AutomationActivities.ps1** — Use parameters: `-DbPath`, optional `-SqliteExePath` (path to sqlite3.exe), `-WindowMinutes` for a time-window filter.
- **Get-AutomationActivitiesByDays.ps1** — Use parameters: `-Days` (1–90, default 30), `-DbPath` (default `C:\RMM\Activities.db`), optional `-SqliteExePath`, plus NinjaOne credentials (same as Get-AutomationActivities). Uses the same DB and `INSERT OR IGNORE` so no duplicate activities are created.
- **Invoke-ScriptStatusSync.ps1** — Use script parameters (e.g. `-DbPath`, optional `-SqliteExePath`, `-CustomFieldName`, `-ForceRebuild`, `-LookbackDays`, `-PreserveExisting`, `-TrimRunPrefix`). Defaults match the original behavior; override when calling the script.
- **Invoke-ScriptTracker.ps1** — Use parameters: `-DbPath` (default `C:\RMM\Activities.db`), optional `-SqliteExePath`, `-BaseOutputFolder` (default `C:\RMM\Reports\Script Tracking`), `-KbArticleIdMapPath` (default: `BaseOutputFolder\.kb-article-ids.json`; omit or leave empty to skip KB deep links), `-NinjaOneInstance`, `-NinjaOneClientId`, `-NinjaOneClientSecret`. Credentials via Ninja-Property-Get (custom fields) when run in NinjaOne, or env vars / parameters.
- **Update-ScriptTrackerKbArticleMap.ps1** — Use parameters: `-BaseOutputFolder` (where to write `.kb-article-ids.json`), `-DestinationFolderPath` (KB folder to enumerate, default "Script Tracking"), `-MergeWithExisting` (default true), plus NinjaOne credentials (same as Invoke-ScriptTracker).
- **Publish-ScriptTrackerToKnowledgeBase.ps1** — Use parameters: `-BaseOutputFolder` (same as Invoke-ScriptTracker output folder), `-DestinationFolderPath` (KB folder for new articles, default "Script Tracking"), `-SkipUnchanged` (default true), `-ForceUpdate`, `-WhatIf`. Credentials via Ninja-Property-Get / env / parameters (same as Invoke-ScriptTracker).
- **Invoke-AutomationDashboard.ps1** — Uses `$ReportMonth` and `$URLOverride` (or env vars `reportMonth`, `URLOverride`). Credentials via Ninja-Property-Get.
- **Invoke-ActivitiesDbMaintenance.ps1** — Use parameters: `-DbPath` (default `C:\RMM\Activities.db`), `-RetentionDays` (default 90; set 0 to skip deletion), `-SqliteExePath`, `-Vacuum` (optional, to reclaim space after delete).

All scripts are standalone: no shared config files or helper modules between them.
